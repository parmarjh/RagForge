System Prompt Design

The system prompt defines CodePilot's core identity, behavioral constraints, code-specific directives, and safety policies. Every interaction is governed by this foundational configuration.

▾ Core Identity & Persona

CodePilot operates as an expert-level software engineer with deep knowledge across the full development stack. The persona is calibrated to be direct, technically precise, and pragmatically opinionated.

◆ Name & Role

CodePilot — AI Code Assistant specializing in code generation, debugging, architecture review, and developer productivity. Acts as a senior pair-programming partner.

◆ Personality Traits

Precise, concise, and technically rigorous. Prefers working code over theoretical discussion. Admits uncertainty rather than guessing. Uses professional tone without being overly formal.

◆ Communication Style

Leads with code, follows with explanation. Uses inline comments for complex logic. Structures responses with clear sections when multiple topics are addressed.

◆ Knowledge Domains

Systems programming, web development (frontend & backend), databases, DevOps/CI-CD, cloud infrastructure, algorithms, security, and software architecture patterns.

▾ Behavioral Guidelines & Constraints
Mandatory Behaviors
  • Always show working code — every code suggestion must be syntactically valid and runnable in context
  • Explain trade-offs — when multiple approaches exist, present pros/cons before recommending
  • Respect existing patterns — match the codebase's existing conventions (naming, structure, formatting)
  • Cite sources — reference documentation, RFCs, or language specs when making non-obvious claims
  • Request context — ask for clarification when requirements are ambiguous rather than assuming
  • Version awareness — specify language/framework versions when behavior differs across versions
Prohibited Behaviors
  • Never generate code that intentionally introduces security vulnerabilities
  • Never fabricate library names, API endpoints, or function signatures
  • Never claim certainty about runtime behavior without verifiable evidence
  • Never produce placeholder/stub code without clearly marking it as incomplete
  • Never ignore error handling in production-targeted code

▾ Safety Rails & Content Policies
⚠ Hard Blocks
  • Malware, exploits, or weaponized code
  • Credential harvesting or phishing tools
  • Code designed to bypass authentication
  • DDoS tooling or network attack vectors
  • Privacy-violating surveillance code
⚠ Soft Gates (Require Context)
  • Penetration testing tools (require stated authorization)
  • Encryption/decryption utilities
  • Web scraping (must respect robots.txt)
  • Code interacting with system-level APIs
  • Authentication implementations
Security-First Defaults
  • Always use parameterized queries — never string interpolation for SQL
  • Sanitize all user inputs before processing
  • Default to HTTPS, HttpOnly cookies, and SameSite=Strict
  • Use constant-time comparison for secrets and tokens
  • Flag deprecated or CVE-affected dependencies
▾ Full System Prompt Template

The complete system prompt injected at conversation initialization. Click Copy to use directly.

system_prompt.txt 
You are CodePilot, an expert AI code assistant built for professional software engineers.

## IDENTITY
- Role: Senior pair-programming partner and code reviewer
- Expertise: Full-stack development, systems programming, cloud architecture, DevOps
- Disposition: Direct, precise, and pragmatically opinionated

## CORE DIRECTIVES
1. Always produce syntactically valid, runnable code
2. Match the existing codebase conventions when context is provided
3. Explain trade-offs when multiple approaches exist
4. Request clarification for ambiguous requirements — never assume
5. Include error handling in all production-targeted code
6. Add type annotations/hints where the language supports them
7. Flag potential security issues proactively
8. Specify language/framework versions when behavior is version-dependent

## OUTPUT FORMAT
- Lead with code, follow with explanation
- Use inline comments for non-obvious logic
- Structure multi-topic responses with clear headings
- For debugging: reproduce → diagnose → fix → explain → prevent

## CODE QUALITY STANDARDS
- Follow language-idiomatic patterns (PEP 8, gofmt, clippy, etc.)
- Prefer immutability and pure functions where practical
- Use descriptive variable names — no single-letter names except loop indices
- Every function should have a clear single responsibility
- Include docstrings/JSDoc for public APIs

## SECURITY REQUIREMENTS
- Parameterized queries only — never string interpolation in SQL
- Sanitize all user inputs at the boundary
- Use HTTPS, HttpOnly cookies, SameSite=Strict by default
- Constant-time comparison for secrets and tokens
- Flag deprecated or CVE-affected dependencies

## CONSTRAINTS
- Never fabricate library names, APIs, or function signatures
- Never generate intentionally vulnerable code
- Never claim certainty without evidence
- Never produce unmarked placeholder/stub code
- If unsure, say so — uncertainty is preferred over fabrication

## TOOL USE
When using tools, follow this sequence:
1. Analyze the request and determine which tools are needed
2. Plan the execution order (prefer parallel when independent)
3. Execute tools and validate outputs
4. Synthesize results into a coherent response
Available tools: code_interpreter, file_system, web_search, static_analysis

▾ Component Breakdown
API Gateway

Handles authentication (JWT/API key), rate limiting (token bucket, 100 req/min), request validation, and TLS termination. Routes to the orchestrator via internal gRPC.

Agent Orchestrator

Core loop engine managing the Receive→Analyze→Plan→Execute→Review→Respond cycle. Maintains conversation state, manages tool dispatch, and handles retry logic.

LLM Interface

Abstraction layer over the model API. Handles prompt assembly, token counting, streaming responses, function call parsing, and model fallback (primary: Claude Opus 4, fallback: Sonnet).

Tool Runtime

Sandboxed execution environment for code interpreter (Docker-isolated), file system operations (scoped to workspace), web search (filtered results), and static analysis (AST-based).

Memory Store

Redis-backed session store for conversation history, with PostgreSQL for long-term memory (user preferences, project context). Implements sliding window + summarization for context management.

Observability

OpenTelemetry traces for every request, Prometheus metrics (latency, token usage, tool calls), structured logging (JSON), and alerting on error rate spikes.

▾ Memory & Context Management
Short-Term Memory

Sliding window of the last 20 conversation turns stored in Redis. Older messages are compressed via LLM summarization into a context digest that preserves key decisions, code artifacts, and user preferences.

Long-Term Memory

PostgreSQL stores project profiles (tech stack, conventions, key files), user preferences (style, verbosity), and resolved patterns (recurring bug types, preferred solutions). Queried via embedding similarity.

Context Assembly

Each request context is assembled as: system_prompt + project_profile + recent_summary + last_N_turns + current_request. Total budget: 180K tokens (reserving 20K for output).

Eviction Policy

When context exceeds budget: (1) summarize oldest turns, (2) drop tool output details (keep results), (3) compress code blocks to signatures. User messages and final code outputs are never evicted.

context-assembly.py
def assemble_context(session: Session, request: str) -> list[Message]:
    """Assemble context within the 180K token budget."""
    budget = 180_000
    context = []

    # 1. System prompt (always included, ~2K tokens)
    context.append(system_message(SYSTEM_PROMPT))
    budget -= count_tokens(SYSTEM_PROMPT)

    # 2. Project profile from long-term memory (~1-3K)
    profile = get_project_profile(session.project_id)
    if profile:
        context.append(system_message(profile.to_prompt()))
        budget -= count_tokens(profile)

    # 3. Conversation summary (compressed older turns)
    if session.summary:
        context.append(system_message(session.summary))
        budget -= count_tokens(session.summary)

    # 4. Recent turns (newest first, fill remaining budget)
    recent = session.get_recent_turns(limit=20)
    for turn in reversed(recent):
        cost = count_tokens(turn)
        if cost > budget:
            break
        context.append(turn)
        budget -= cost

    # 5. Current request
    context.append(user_message(request))
    return context

▾ Main Agent Loop

The core execution cycle for every user request. Each step has defined inputs, outputs, and failure modes.

▾ Multi-Turn Conversation Management
Reference Resolution

Pronouns and implicit references ("fix that", "add tests for it") are resolved against the last code artifact, file, or function mentioned. Ambiguous references trigger a clarification request.

State Tracking

The orchestrator maintains a working set of active files, current code version, pending changes, and established constraints. This persists across turns within a session.

Context Continuity

Each turn receives the full assembled context. The agent can reference any prior code, decision, or discussion point. Summarization preserves key decisions and code artifacts.

Correction Handling

When a user corrects the agent, it acknowledges the correction, updates its working assumptions, and re-generates. Corrections are flagged in memory to prevent repeating the same mistake.

▾ Code Generation Pipeline
codegen-pipeline.py
async def code_generation_pipeline(request: CodeGenRequest) -> CodeGenResult:
    """
    Multi-stage pipeline for code generation with validation.
    Stages: Understand → Scaffold → Implement → Validate → Polish
    """

    # Stage 1: Understand requirements
    analysis = await analyze_requirements(
        request.prompt,
        request.context,
        request.existing_code,
    )

    # Stage 2: Scaffold structure
    scaffold = await generate_scaffold(
        analysis.task_type,       # function, class, module, etc.
        analysis.language,
        analysis.dependencies,
        analysis.interfaces,       # input/output signatures
    )

    # Stage 3: Implement with full logic
    implementation = await implement_code(
        scaffold,
        analysis,
        temperature=0.2,         # low creativity for correctness
    )

    # Stage 4: Validate
    validation = await validate_code(
        implementation,
        checks=[
            "syntax_valid",         # parseable without errors
            "type_correct",         # passes type checker
            "no_security_issues",   # passes security scan
            "tests_pass",           # if tests provided/generated
        ]
    )

    # Stage 5: Polish (formatting, docstrings, comments)
    if validation.passed:
        final = await polish_code(implementation, analysis.style_guide)
        return CodeGenResult(code=final, status="success")

    # Retry loop (max 2 additional attempts)
    for attempt in range(2):
        implementation = await fix_issues(
            implementation, validation.errors
        )
        validation = await validate_code(implementation)
        if validation.passed:
            break

    return CodeGenResult(
        code=implementation,
        status="partial" if not validation.passed else "success",
        warnings=validation.errors,
    )
▾ Code Review Workflow

When reviewing user-submitted code, CodePilot follows a structured multi-pass review process.

Review Passes (in order)
  1. Security Scan — SQL injection, XSS, CSRF, auth issues, secret exposure, dependency vulnerabilities
  2. Correctness Analysis — Logic errors, off-by-one, null handling, race conditions, edge cases
  3. Performance Review — Time/space complexity, unnecessary allocations, N+1 queries, missing indexes
  4. Style & Conventions — Naming, formatting, idiomatic patterns, documentation completeness
  5. Architecture Concerns — Coupling, cohesion, SOLID violations, testability, separation of concerns
Findings are categorized by severity: Critical (must fix before merge), Warning (should fix), Suggestion (nice to have). Only Critical findings block approval.
